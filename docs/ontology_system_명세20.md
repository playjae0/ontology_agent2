## 1. 목적과 범위 [확정]

- 사내에서 지속 운영·확장 가능한 온톨로지/GraphDB 시스템을 만든다. 공정정보층에서 시작하여 정보층을 추가하고, 층간 연결로 지식을 확장한다.
- 제조정보 특성상 **관계성이 1급 시민**이다. 문서는 Excel(밀도 높은 공정 spec/관리정보, 2차원 테이블)과 PPT(글+이미지 복합 보고자료)가 다수이며 Word/PDF가 보조.
- 데이터는 사내 반출 불가 → **파서는 사내 구현**. 외부에서 설계하는 것은 파서 출력(JSON)을 소비하는 온톨로지 에이전트와 플랫폼.
- 모든 결합은 **파일 계약(JSON)** 으로만 한다. 파서↔에이전트↔플랫폼은 코드가 아니라 JSON으로 만난다.

### 1.1 PoC 범위 [확정]

**공정층 구축 + 품질지식층 구축 + 층간 연결까지.**
- 근거: Rule of Three(§3.3)상 2번째 층이 실재해야 C의 공통 경계와 층간 연결(§8)의 실물 답이 관찰된다. PoC가 데모이자 설계 검증 실험을 겸한다.
- 2번째 층 = **품질지식층**: PFMEA류가 담는 잠재 고장모드/원인/영향의 정적 지식 체계. 실제 발생 사건을 담는 **불량이력층**(사건 스트림, 시점 보유)과 성격이 다르므로 구분 — 불량이력층은 3번째 층 후보로 로드맵 유지. 자재/BOM층도 후보(§5.4 관찰 항목).
- PoC 공정층은 조립 대공정만 심는다 — 전극/화성은 seed 키 추가로 확장(§5.2).
- 질의 지원 범위: 질문 유형 Q1~Q5·Q8(§5.6.5). Q6은 훅만, Q7은 범위 밖.

## 3. 범용성 확보 전략 [확정]

### 3.1 문제

층(공정정보/품질지식/불량이력/설비이력...)을 여러 개 만든다. 새 층을 추가할 때 **손대는 지점을 최소화**(이상적으로 층 폴더엔 코드 0, config+스키마만)하는 것이 확장의 핵심 요구다. 이를 위해 "층마다 다른 부분"을 최대한 **값(config 데이터)** 으로 내리고, core는 그 값을 **개수·이름·유무를 가정하지 않고 순회·해석**한다(§3.6). 남는 진짜 "절차 차이"만이 마지막 판단 대상이며, 표본 2개(공정·품질) 설계에서 관찰한 결과 그 차이는 대부분 config 결정점으로 표현 가능함이 드러났다(묶음3, v1.8).

### 3.2 세 겹 경계

```
[ A ] 모든 층이 공유 — core에 한 번 짜고 안 건드림
      id 발급 · 동의어 사전 · 저장 · 그래프 탐색 · LLM 호출 · 임베딩 · role 핸들러 루프
      · 범용 build(쓰기 파이프라인) · 범용 query(읽기 파이프라인) · 범용 skeleton-planter(골격 심기)

[ B ] 층마다 "값"만 다름 — 층별 config + 스키마 파일
      카테고리(정의문) · 관계 목록 · 프롬프트 · (카테고리쌍→관계) 매핑 · 문장화 템플릿 ·
      query_traverse/cross_layer_traverse · mirrors 적용 여부 · skeleton 데이터(형태+내용) · doc_type 스키마

[ C ] 층마다 "동작"이 다를 수 있는 것 — 원칙적으로 없음(config 결정점으로 흡수)
      골격 심기·쓰기·읽기 절차는 core 범용 파이프라인이 B의 결정점을 읽어 구동.
      config로 표현 안 되는 절차가 발견되면 → 층별 오버라이드 코드가 아니라 명시적 실패(§3.6).
```

- A는 core에 둔다. **v1.8: build/query/skeleton도 A로 승격** — 표본 2개(공정·품질) 설계에서 이 절차들의 층별 차이가 전부 B(config 결정점: skeleton.type, mirrors.enabled, query_traverse 등)로 표현됨을 관찰했기 때문(추측이 아니라 실측 — §3.3).
- B는 "값"이라 리스크 없이 층별 파일로. core는 이 값들을 **개수·이름·유무 무관하게** 순회(§3.6).
- C는 이상적으로 비어야 한다. 남으면 그것은 "config 표현력 부족"의 신호이며 core 패턴 추가로 해소(§3.6 탈출구).

### 3.3 C의 일반화 시점: Rule of Three [확정]

C의 공통 뼈대는 표본이 둘 이상 있어야 관찰로 뽑힌다. 하나에서 뽑은 "공통점"은 그 하나의 특징일 뿐이며(사례 하나로는 일반화가 정의상 성립하지 않음), 구조층과 사건이력층은 성격이 달라 예측이 빗나갈 확률이 높다.

- **1번째 층(공정)**: C를 층 전용으로 짠다. 일반화 시도 안 함.
- **2번째 층(품질지식)**: 공정층 C를 복사해 옆에 두고 고쳐 짠다. "안 고친 곳=공용 / 새로 짠 곳=층별"이 **결과로 드러난다.**
- **관찰된 만큼 승격 (v1.8 명확화)**: Rule of Three는 "3층까지 아무것도 하지 마라"가 아니라 "**추측으로 미리 짜지 마라, 관찰된 만큼만 잡아라**"다. 표본 2개를 실제 설계까지 마친 상태에서 절차가 명백히 공통이면 2에서 A로 올려도 된다(build/query/skeleton이 그 경우). "3"은 안전마진이며, 아직 관찰 못 한 나머지를 위한 것 — 지금 잡을 수 있는 것까지 미루는 게 아니다.
- **과설계와의 경계**: 만능 엔진(과설계)은 "표본 없이 범용 엔진을 추측으로 짜는 것". 우리는 2개 층의 실측 공통을 올리고, 못 관찰한 절차는 명시적 실패로 드러내 미룬다(§3.6) — 추측 코드가 아니다.

### 3.4 격리 규칙 (미리 만들지 말고, 자리만 표시) [확정]

- **(가) 미래 코드 선(先)작성 — 금지.** 아직 안 만든 층을 위한 처리(예: causes 처리)를 미리 넣지 않는다. 맞을지 모르는 코드는 짐이 된다.
- **(나) 바뀔 자리 격리 — 필수.** C를 공용(A) 파일에 섞지 않고 층 전용 위치에 모아 이름표를 붙인다. 코드를 더 만드는 게 아니라 위치로 표시만 하는 것.
- (나)가 "나중에 왜/어디가 안 맞는지 보고 고친다"의 전제조건 — C가 A에 스며들면 진단이 불가능해진다.

### 3.5 코드 배치 [확정]

```
ontology/
├── core/                    A — 층 이름·필드명·관계명이 한 글자도 등장하지 않음
│   ├── graph.py                 노드/엣지 추가, id 발급, 저장/로드, neighbors(ids, traverse_spec←인자)
│   ├── dictionary.py            동의어 사전: register / lookup
│   ├── matcher.py               개체 판정 호출
│   ├── llm.py                   게이트웨이 호출 + JSON 파싱
│   ├── ingest.py                role 핸들러 루프 (§6.6)
│   ├── build.py                 범용 쓰기 파이프라인 (config+스키마 구동, 2-pass·재인입·mirrors)
│   ├── query.py                 범용 읽기 파이프라인 (config 구동, 링킹·확장·수집·답변)
│   └── skeleton.py              범용 골격 심기 (config skeleton 데이터: type=tree|flat + data)
├── layers/<층이름>/          **config + 스키마만 (코드 0)**
│   └── config.json              B 일체: 카테고리·관계·프롬프트·매핑·템플릿·traverse·mirrors·skeleton
├── schemas/                 doc_type별 스키마 + blocks (role_어휘_정의서 기준)
├── router.py                층 폴더 자동 발견 (등록 코드 없음)
└── data/<층이름>/            산출물 (nodes+edges JSON = 진실, 청크·사전·큐는 공유 §8-R)
```

- **층 추가 = config.json + 스키마 파일.** 층 폴더에 코드 0. router가 폴더를 자동 발견하므로 등록 배선도 없음.
- **성공 판정**: 2번째 층(품질)이 config만으로 core 범용 파이프라인에서 도는가(구현 단계3에서 실측 — §16/§구현문서 §7).
- 규모 감각 (참고치 — 구속 제약 아님): core 파일당 수십 줄, 층당 100~200줄, 층 3개 기준 수백~1천 줄 내외.
  **구속력 있는 기준은 줄 수가 아니라 P3(사람이 전부 읽고 파악 가능)와 "층 추가 시 core 무수정"이다.**
  줄 수를 맞추기 위한 압축·생략(에러 처리, 로깅, provenance 기록, 명시적 변수명의 희생)은 금지 — 명료함이 짧음보다 우선.
- 간단함의 근원 (v1.8 정정): core가 단순한 이유는 절차를 층 폴더로 밀어냈기 때문이 아니라, **개수·이름·경우를 분기하는 코드가 없기 때문**이다(§3.6). core는 config를 순회할 뿐이라, "모든 층을 수용하는 똑똑한 실행기"가 아니라 "가정을 안 하는 단순한 순회기"다 — 이것이 과설계(만능 엔진)와 정반대인 이유.
- 유지보수: 관리 지점이 core 한곳으로 모인다. 층 폴더엔 config만 있으므로, 문제가 "값(config) 문제"인지 "패턴(core) 문제"인지가 명확히 갈린다.

### 3.6 범용성 원리 [확정 — v1.8]

**core 코드는 층별 축의 개수·이름·유무를 가정하지 않는다.** role이 "항목 명칭이 달라도 역할(role)이 같으면 같은 코드로 흐르게 한 구조"였듯, 이 원리를 **모든 층별 축**(관계·카테고리·골격 패턴·확장 스펙·mirrors 등)으로 일반화한다.

- **개수 무가정**: 관계가 3개든 N개든 config의 relations 리스트를 순회. "관계는 3종"을 코드에 박지 않는다 — 다음 층이 5종이어도 그냥 돈다. 안 쓰는 축은 config에서 빠져 있을 뿐(놀게 둠), 코드는 있는 만큼만 처리.
- **이름 무가정**: "part_of"·"Process" 등 층 어휘를 코드에 쓰지 않는다(§0-1). config가 준 이름으로 동작.
- **유무 무가정**: mirrors 없는 층, precedes 없는 층도 그 항목이 config에 없을 뿐 — core는 깨지지 않는다.

**§3.4-(가)와의 관계 (선작성 아님)**: (가)가 금지하는 것은 "미래 층의 **특정 동작**을 추측으로 구현"하는 것(예: 불량이력층의 causes 처리 로직을 미리 짜기). 반면 "개수·이름·유무를 가정하지 않는 순회 구조"는 **미래 축의 정체를 전혀 모르는 채로** 무엇이 오든 수용하는 것 — 가정이 0이므로 (가)의 정반대이며 장려된다. "엣지를 4개 이상으로 열어둔다"는 미래 엣지를 미리 만드는 게 아니라, config에 있는 것만 쓰고 없으면 안 쓰는 것.

**탈출구 = 오버라이드 코드가 아니라 명시적 실패**: config로 표현 안 되는 층별 절차가 나오면, 층 폴더에 오버라이드 코드를 두지 않는다(특수 코드가 층마다 흩어지면 관리가 무너짐). 대신 core 범용 파이프라인이 **명시적으로 실패**하며 "이 결정점은 config로 표현 안 됨 → core에 패턴 추가 필요"를 알린다. 확장은 항상 core에서 일어나므로(모든 층이 혜택), 관리 지점이 core 한곳으로 일원화된다. 이것이 Rule of Three의 "3층에서 안 맞으면 승격"을 실무적으로 대체 — 안 맞는 것이 조용히 뭉개지지 않고 시끄럽게 드러난다.

### 5.2 칸2 — 골격

- **골격 = 대표 프로세스의 공정(기능) 트리.** 공정="무엇을 한다"(기능: 타발한다), 설비="무엇으로 한다"(물리: 노칭 프레스). 지금은 1:1처럼 보여도 변하는 속도가 다르다 — 공정(기능)은 수년 단위 안정·공장 공통, 설비는 공장·라인·호기·기종·증설로 변동. 골격 자격은 안정적·공장공통인 것에만 있다("골격은 사람이 고정"이 유지되려면).
- **공장 차원 미도입.** 근거(문서 실태): 문서 거의 전부가 공장 무관 서술이며 설비 관련 문서만 일부 상이 → 그래프에 담길 지식 자체가 공장 무관. 공장·호기 구분("A공장 3호기")은 설비이력층 소관이며 "기종(공정층)↔개체 호기(설비이력층)" 연결은 §8 안건. 공장 차이의 대부분이 설비 차이이므로, 설비를 골격에서 빼는 순간 골격은 공장 공통 유지가 가능해진다. 필요해지면 site 좌표를 공통 필드에 추가(L1급).
- **전극/조립/화성은 단일 층.** 층 분리 기준은 구역이 아니라 정보의 성격 — 셋 다 같은 카테고리·관계·스키마로 처리되는 공정 구조 정보. 최상위 형제 3개로 표현되고 대공정 간 순서는 최상위 형제의 seed 순서로 자연 표현. PoC는 조립만 심는다.
- **가변 깊이 허용, 시작은 2단(대공정+세부공정)으로 얇게.** 그래프에서 트리는 노드+part_of일 뿐 깊이 개념이 코드에 없다(고정이 오히려 검증 코드를 늘림). 하위공정은 문서가 그 해상도로 말하는 것이 관찰될 때 노드 추가 — 가변 깊이+id 불변이라 기존 그래프 무영향. PoC 첫 골격 = 조립+세부공정 6개 = 노드 7개.
- **절단 기준(골격 vs entity)**: "설비가 다른 기종으로 바뀌어도 이 단계는 존재하는가?" 예 = 공정(골격 후보, 예: 노칭 타발) / 아니오 = 설비 동작(entity, 예: 서보 하사점 보정).
- **골격 작성 규칙**: canonical 이름은 층 내 유일(유일 불가 시 부모 접두 "스태킹-탭정렬") — 가변 깊이에서 anchor 조회 모호성을 막는 유일한 필요 규칙. 형제 간 순서는 seed 나열 순서.
- **process_no는 attribute로 강등** — 사내 공정번호 체계가 비표준·가변이므로 조회 키 불가(변하는 값을 키로 쓰면 P4 실질 위반). anchor 조회 키 = 이름(canonical+alias 사전). 번호는 노드에 참고 저장만.
- 설비·기구부·지그·금형 = entity(발견 대상). 골격에 넣지 않는다.
- **극성(cathode/anode) 처리 (v1.4 — 사내 대조 반영)**: 극성이 갈리는 **레벨이 공정마다 다르다**. ①스택 이전 대부분: 공정 flow는 극성 무관 **단일**(precedes 한 벌 = 대표 흐름), 극성은 **설비·인자(잎사귀)에서** 갈림 — 같은 Process에 cathode/anode Unit·Property가 각자 part_of, 설비가 물리적으로 분리되고(음/양극은 스택 전 접촉 금지) spec도 극성별로 다름. ②탭 용접 등 극성 잔존 공정: **공정 자체가 극성별**(cathode laser welding처럼 canonical에 극성 포함) → Process 레벨에서 갈림. **흐름은 처음부터 끝까지 단일 스트림 — 갈라지지 않는다.** 스택 이전 공정은 노드 단일(극성은 잎사귀), 극성 잔존 공정은 음극 용접·양극 용접이 precedes로 **순차**(병행 아님)로 이어질 뿐 스트림은 하나. 극성 잔존 공정 쌍에는 mirrors 엣지를 얹어 대칭 표시(§5.3, precedes와 공존). flow 질의는 이 단일 스트림을 답한다. 갈림·합류 지점은 골격 seed가 사람 지식으로 표현(Tier1). 극성별 노드의 canonical은 극성 포함(사내 관행), 표면형(극성 제거)은 alias 공유. **entity 해소 규칙 (v1.7 정밀화)**: 극성 결합 canonical은 **①category ∈ {Unit, Property} AND ②행의 electrode_type이 cathode/anode로 확정**일 때만 적용("cathode 노칭 프레스"). Failure(품질층) 등 그 외 카테고리는 극성 결합하지 않음 — 불량 현상은 극성이 아니라 공정 위치(occurs_in)로 정체가 정해지고, 극성 결합 시 causes 병합 이득이 소실됨. 같은 Failure가 cathode행·anode행에 걸치면 한 노드로 병합되고 occurs_in 엣지가 극성별 공정으로 각각 생성. **행 극성이 both/무표기이면 Unit/Property도 극성 무관 노드로 생성** — 나중에 극성별 상이가 드러나면(극성 무관 Property에 극성별 다른 spec이 쌓여 spec_conflict 발생) 그 conflict가 **극성 분리 신호**가 되어 수정 도구 node split으로 사람이 가름(자동 커밋+사후 수정, "되돌리기 쉬운 쪽"). control_item류가 실물에서 극성을 항상 명시하는지는 [사내 확인] — 명시 확실 시 파서 계약으로 극성 필수화 가능.
- **게이팅 조건 ③ (v1.12, F2 마감)**: 위 ①②에 더해 **③인입 문서의 층 config가 polarity를 선언**할 때만 극성 결합. 걸침 필드(PFMEA control_item→process)는 문서 층(quality)이 polarity 미선언이므로 **비극성 생성** — 걸침 문서의 극성 표기 신뢰성이 실물 확인 전이라 보수적 기본값(위 [사내 확인] 후 "기록 대상 층 기준" 재검토 여지).
- **Property canonical = 세부공정 스코프 (v1.12 F4 → v1.14 G1 정본 교정)**: Property canonical은 **`{부착 부모 canonical}::{인자 표면형}`**이며 **부모의 기본은 세부 공정(@process_ref)**이다(설비 아님). 예: `"노칭::노칭 정밀도"`, 극성 시 `"노칭::cathode 노칭 정밀도"`(극성은 Property 표면형에 결합). 근거: 관리항목은 **세부 공정 단위**로 관리(tab welding의 "초음파 용접::용접 강도" vs "pre vision::검사 정밀도"). 설비 단위는 과분할이며, 걸침 필드(PFMEA control_item, 설비 attach 없음)가 좌표 폴백이라 CP `노칭 프레스::` vs PFMEA `노칭::`로 키가 갈려 규칙B 보강 병합이 깨진다 — 세부공정 스코프로 통일해야 CP·PFMEA가 같은 키로 병합. 같은 인자명이라도 세부 공정이 다르면 다른 노드(§9). 표면형은 alias 공유. 스코프 대상 카테고리는 config `canonical_scope`(값). **경고**: `attach_to_field:"설비"`로 설비 스코프 전환은 L1로 가능하나 걸침 필드 병합 규칙(설비 미해소 시 좌표 조상 검사)을 **선행 마감**해야 함("L1 한 줄이 지뢰").
- **좌표 불명 오병합 방지 (v1.14, G4)**: 부모(process_ref)가 미해소여서 무접두 canonical로 생성되는 Property는 후보 조회에서 **스코프 canonical 보유 노드를 제외**(무접두끼리만 병합). 근거: 좌표 불명 행이 동명 스코프 노드에 alias 히트해 오병합되면 복구 불가(§9). 동명 존재 여부로 병합/분리가 갈리는 비결정성 제거. 실데이터 파일럿 전 필수(현 mock 미발현).
- **골격 파일은 파서와 에이전트의 공유 자산** — 좌표 태깅용 닫힌 목록(canonical+alias)을 파서에 공급(§12).

### 5.3 칸3 — 관계 목록

| 관계 | 의미 | 연결 | 출처 |
|---|---|---|---|
| `part_of` | 계층 (A는 B의 일부) | Process→Process, Unit→Process | seed + entity 부착 |
| `precedes` | 순서 (A 다음 B) | Process→Process 형제 간 | seed 나열 순서만 |
| `has_property` | 관리인자 보유 | Process/Unit→Property | edges 선언·(카테고리쌍→관계) 매핑 |
| `mirrors` | 극성 대칭 (설비 구조 대칭, 값 비공유) | Process↔Process, Unit↔Unit, Property↔Property | **자동 규칙(아래)** |

**mirrors 자동 연결 규칙 (v1.4)**: 두 노드가 ①같은 카테고리 ②electrode_type만 반대(cathode↔anode) ③극성 토큰 제거 후 canonical 동일(문자열 비교 — LLM 불요. **극성 결합 주체는 에이전트**(v1.12, F11), 결정성 근거는 파서의 electrode_type 구조 필드. 파서는 표면형에 극성 미결합 — §12) ④[Process면] 같은 precedes 위치 / [Unit·Property면] 같은 부모 부착 → mirrors 엣지 생성. 이어서 **양쪽 자식(part_of/has_property) 수·구성 비교 → 불일치 시 `mirror_asymmetry` 큐**(대칭 선언 후 짝 없는 항목 노출 — "문서 누락 vs 진짜 극성 차이" 검토. 한쪽 극성 문서만 갱신되는 실무 사고의 안전망). mirror는 대칭을 **묶지 않고 노출**만 함 — 값·spec 공유 없음.
- **self-heal (v1.11)**: mirror 규칙은 매 build마다 **재평가**한다. 큐 항목은 (극성 제거 canonical, 부모) **쌍 키로 dedup**(재빌드·재인입에도 중복 큐잉 없음). 재인입·후속 문서로 대칭이 회복되면(양쪽 자식 수·구성 일치) 해당 mirror_asymmetry 항목을 **큐에서 제거**한다. "첫 감지만 큐잉"으로 고정하면 안 됨 — 해소를 스스로 반영 못 하므로(구현 단위 1c 검수에서 드러난 결함).

경계 규칙:
- **공정층에 causes/affects 미도입** — 고장모드·원인은 구조가 아니라 품질지식층 소관. 관계 목록의 봉쇄가 층 정체성의 실질이다.
- **precedes는 세부공정 형제 간에만, 출처는 seed 나열 순서 하나로 고정** — 문서에서 순서 추출 금지(문서별 서술 차이로 인한 오염 방지).

### 5.4 칸4 — 카테고리 정의문·프롬프트

**카테고리 정의문** (소비자는 추출·판정 LLM — "무엇인가"보다 "무엇과 헷갈리는가"를 서술. 정의문이 판단 품질을 좌우하며 B(config)에 위치):

> **Process(공정)** — 제품을 만들기 위해 수행하는 작업 단계. 기능으로 정의되며 특정 설비 기종이 바뀌어도 존재한다(예: 노칭, 스태킹). 설비의 이름이 아니라 "하는 일"의 이름.
>
> **Unit(설비)** — 공정을 수행하는 물리적 장비의 기종/표준 명칭(예: 노칭 프레스, 스태커). 지그·금형 등 도구류 포함(part_of로 부착). 호기 번호·특정 공장 개체("3호기", "A공장 노칭기")는 이 층에서 다루지 않는다.
>
> **Property(관리인자)** — 공정 또는 설비에서 관리·측정·통제되는 항목의 이름(예: 노칭 정밀도, 금형 클리어런스). 사내에서 관리항목/특성/CTQ 등으로 불리는 것 포함. 인자의 이름만 해당 — 규격값(±0.1mm)·측정값·판정결과는 Property가 아니라 attribute.

- **자재(전극·분리막·전해액 등): 카테고리 미도입** — 공정을 따라 흐르며 변형되는 제3 성격(구조도 사건도 아님). 별도 층(자재/BOM층) 후보. PoC에서 자재 열은 content/meta로 배정하고 **등장 빈도를 관찰 항목**으로 기록 → 3번째 층 우선순위(불량이력 vs 자재) 판단 재료.

**프롬프트 3종**:
1. **추출(prose 전용)**: 정의문 3종을 프롬프트에 삽입. 규칙 — "이 청크가 주제로서 서술하는 개체만"(스친 언급·일반명사·수치·날짜·판정결과 제외) + "카테고리는 정의 중 하나만, 어디에도 안 맞으면 추출하지 않음" + "애매하면 빼는 쪽". 출력 `{"mentions":[{"surface","category"}]}`.
2. **개체 판정**: "글자 유사성이 아니라 의미·문맥으로 판단" + 표기 변형 안내(띄어쓰기·영문혼용·축약은 같은 개념 가능성 높음, 다른 대상이면 글자가 비슷해도 다른 개념) + **비대칭 기준 명시**("확신 없으면 match가 아니라 uncertain — 잘못된 병합이 잘못된 신규보다 해롭다"). 입력 = mention + 후보들(canonical, aliases, 부착 위치). 출력 `{"type":"match|new|uncertain","matched_id","confidence"}`.
3. **좌표 태깅(파서 소관, 계약상 여기서 정의)**: 골격 세부공정 닫힌 목록(canonical+alias) 제공 → "이 청크/시트가 다루는 공정 선택, 없으면 null".

**추출-판정 카테고리 정책**: 추출이 분류까지 수행(호출 1회, "전자" 방식). 안전망 — 판정 단계에서 **추출 카테고리와 최상 후보의 카테고리가 불일치하면 match 금지 → 신규 생성 + 수정 큐** (카테고리 오배정이 조용한 오병합으로 이어지는 경로 차단, §9 "되돌리기 쉬운 쪽" 규칙의 적용).

### 5.5 칸5 — 쓰기 절차

1. **문서 내 2-pass**: Pass 1 = 문서 전체 청크/레코드의 추출·판정만 수행(해소 결과를 문서 단위 버퍼에) → Pass 2 = 버퍼를 놓고 부착·edges 생성(이 시점엔 문서 내 등장 개체가 모두 존재). 문서 내 서술 순서 의존 제거 — 부착 실패가 "순서 문제"인지 "진짜 미해소"인지 구분되어 orphan 큐 노이즈 감소.
2. **판정 임계 0.85 단일**: match & confidence≥0.85 → 매칭(alias 누적), 그 외 → 신규+수정 큐. 이중 임계 등 세분화는 보류율 데이터가 생긴 뒤(P7).
3. **재인입 = 문서 전체 교체**: 개정 문서 유입 시 해당 doc_id의 청크·describes·provenance 항목을 걷어내고 재인입. **노드 자체는 삭제하지 않음** — provenance가 여럿인 노드는 다른 문서가 여전히 근거. 해당 문서의 provenance 항목만 제거하고, provenance가 0이 된 auto 노드는 "근거 소멸"로 수정 큐에 표시(자동 삭제도 금지 — 되돌리기 쉬운 쪽). alias도 동일하게 처리(alias provenance가 이를 가능케 함).
   - **엣지 재인입 계약 (v1.3)**: 재인입은 해당 doc_id의 **엣지 provenance도 걷어낸다**. provenance가 0이 된 엣지는 노드와 대칭으로 자동 삭제 금지·evidence_lost 큐 표시.
   - **attribute 재인입 계약 (v1.7)**: attribute 값 항목도 provenance를 보유한다(`{context, value, provenance}` — 노드·엣지·alias와 대칭, §0 불변식). 재인입은 doc_id 단위로 attribute 항목도 회수 → 개정 시 옛 spec 항목이 식별·제거되고(잔존 방지), 같은 문서의 개정값 변경은 spec_conflict가 아니라 교체로 처리(교차출처 충돌 C4와 구별). provenance 0이 된 항목은 evidence_lost 큐.
   - **재인입 회수 규칙 3분류 (v1.11 — 구현 검수에서 정밀화)**: "doc_id 단위 회수"를 대상별로 구분한다(뭉뚱그리면 살아있는 노드가 유령이 됨):
     · **① 회수(제거)**: 그 문서에서 온 청크·describes·엣지 provenance·attribute 항목. provenance-0 → evidence_lost 큐.
     · **② 보존(회수 금지)**: 살아있는 노드(evidence_lost로 남는 노드)의 **사전 엔트리(canonical·alias)는 회수하지 않는다.** 노드가 살아 있으면 그 식별 수단도 살아야 재인입 시 재매칭된다 — 사전 엔트리를 걷으면 사전 미스로 **중복 노드가 생성**된다(단위 1c 검수에서 실측: CP01 재인입 시 cathode 노칭 프레스 2개, mirror 데카르트곱으로 asymmetry 폭증). 사전 엔트리 회수는 노드가 **실제 삭제될 때만**.
     · **③ 재평가**: 그 문서가 유발한 큐 항목(mirror_asymmetry 등)은 재인입 시 **재평가**(§5.3 self-heal) — 해소됐으면 제거, 아니면 갱신. stale로 남지 않는다.
     · **auto_node/uncertain_match sweep (v1.14, G3)**: 이 두 kind는 "문서 발자국"이 아니라 **노드 상태(status=auto/uncertain)의 그림자**다. remove_doc로 걷은 뒤 재인입이 match 경로면 재적재가 안 돼 검토 항목이 증발한다. 따라서 build 말미에 **status=auto/uncertain 노드 전수로 sweep 재작성**(evidence_lost sweep과 같은 패턴) — 큐가 노드 상태의 파생물이 되어 재인입에도 검토 목록이 보존된다. §16.1 "수정 재료를 빠짐없이 기록" 이행.
   - 구현 상태: self-heal(③·§5.3)은 지금 구현. **사전 보존(②)은 노드 유일성 불변식(P4)의 직접 이행이므로 우선순위 상향 — 플랫폼(단위 4) 착수 전 구현**. 근거: 플랫폼의 정상 동작이 문서 재인입이라, ②가 없으면 플랫폼 사용 자체가 그래프를 오염시킴. run.py all 2회 실측(61→147 노드)으로 재인입 경로가 개발 일상에서 이미 발현됨이 확인됨(당초 "실운영 전용"으로 본 판단 수정). 검증: 재인입→노드 수 불변·사전 재매칭·옛 큐 회수.
   - **사람이 수정 도구로 삭제한 엣지는 status="deleted_by_user"로 진실(graph.json)에 provenance와 함께 영속 보존**하며, add_edge는 이 상태의 (src,rel,dst)-by-id를 건너뛴다(재인입 부활 방지). 근거: 사람의 삭제 판단은 문서에서 재생성되지 않는 지식이므로 파생물이 아님(P4/P5) — id 불변이라 by-id 키가 안정. "원본 문서 수정 강제" 대안은 기각(§9 자동커밋·비차단 철학 위반, 원본 편집 불가·해소 아티팩트 오류 시 해결 불가). **enforcement 코드는 엣지 삭제 도구 구현 시점(§13 ②, 국면1 마지막)으로 이연 — 계약·status 값만 지금 확정**(§3.4-(나) 자리 표시). 실사용 스코프는 prose 경로(카테고리쌍→관계) 활성 후가 주(table의 나쁜 엣지는 대부분 노드 오해소=분리 또는 원본 오류).
4. **문서 간 순서 없음**: 2-pass는 문서 안에서만. 교차 문서 의존(문서 A의 인자가 문서 B의 설비에 부착)은 A 인입 시 orphan → **orphan 재시도 배치**(수정 큐의 orphan들을 주기적으로 재해소 시도)로 해결. 인입 순서 강제는 운영을 취약하게 하므로 배제.

### 5.6 칸6 — 읽기 절차

**동작 구조 (4단, LLM은 마지막에만 등장)**: 시스템(코드)이 그래프 연결성으로 근거를 골라 LLM에게 주고, LLM은 받은 근거를 읽고 문장으로 답한다. LLM은 그래프(DB)를 직접 읽지 않는다. (GraphRAG: Graph로 Retrieval, LLM이 Generation. 진실이 JSON이므로 ②③은 파이썬 순회 — Neo4j 뷰 전환 시 같은 일을 Cypher로 표현 가능.)

**5.6.1 링킹 (3단)**
- 1단: 질문 문자열에서 사전(canonical+alias) 표면형 스캔 — 긴 표면형 우선, 무LLM. (구축 때 쌓은 동의어가 질의 히트율로 회수되는 곳)
- 2단(사전 미스): LLM 언급 추출 → 후보검색+판정.
- 3단: [임베딩 청크 직접 검색 — **훅만 확정, 구현 이연** §5.6.6]
- 전 단계 읽기 전용(P6) — 질문 표현을 alias로 누적하지 않음.

**5.6.2 확장**
- `part_of` 하향(자식) 재귀 — "노칭에 대해"의 자연 범위(하위공정·설비 전부).
- `part_of` 상향 1홉(부모만, 맥락용) — 상향 재귀는 전체로 폭발하므로 금지.
- `has_property` 양방향 1홉 (개체↔인자, 역방향 질문 Q5 대응).
- `precedes`는 확장에서 제외 — 옆 공정 청크가 노이즈로 섞이는 것 방지. 순서 정보는 5.6.4 그래프 사실 채널이 담당 (확장="어느 노드의 청크까지 볼까"와 사실 공급="아는 엣지를 말해줄까"는 다른 문제).
- 확장 함수는 따라갈 관계를 인자로 받되, 두 층위를 구분한다: **관계 목록(어느 관계를 따라가나) = B**(config의 query_traverse) / **관계별 탐색 의미(방향·홉·재귀 — 위 part_of·has_property·precedes 규칙) = 현재 C**(층 query 코드). cross-layer 질의(§8)는 목록·의미 추가로 층을 넘는다. 탐색 의미의 config화(B 승격, 관계별 {direction/hops/recursive} 스펙)는 **2번째 층의 탐색 패턴 관찰 후**(Rule of Three) — 지금 제네릭 탐색 엔진을 설계하는 것은 core 복잡화(P3 위반)이므로 이연하고 §3.4-(나)에 따라 자리만 표시.

**5.6.3 수집**
- 2-tier: tier1=직접 링킹 노드의 청크 > tier2=확장 노드의 청크. 상한 8(측정 후 조정). 초과 시 tier2부터 잘림(잘림률 계기판). tier 내 정렬은 문서 최신순. 랭킹 고도화(임베딩 유사도)는 잘림률이 정당화할 때(P7).

**5.6.4 답변 — 이원 근거 채널**
```
LLM 입력 = 질문
  + [그래프 사실]  링킹·확장 노드의 엣지 + 노드 필드를 문장화
                  "노칭 다음 공정은 스태킹" (precedes) / "노칭 프레스는 노칭 공정의 설비" (part_of)
                  "노칭 정밀도의 규격: ±0.1mm (출처: PFMEA-A-003)" (필드) — 맥락형 값은 context별 한 줄씩 렌더("[모델 A] 규격: ..., [모델 B] 규격: ...")
  + [문서 근거]    수집된 청크 원문 (청크ID/문서/섹션 포함)
```
- 근거가 청크 단채널이면 엣지·필드에만 있는 정보(순서·규격)를 답하지 못한다 — 그래프의 구조 자체도 근거여야 함. 엣지·필드→문장 변환은 관계/필드당 템플릿(B, config).
- 프롬프트가 두 채널을 구분해 사용하게 함: 그래프 사실=시스템이 보증하는 구조 정보, 문서 근거=서술 정보. 구조·값 질문은 전자가, 서술 질문은 후자가, 복합은 함께 답을 구성.
- **flow 질의 규칙**: 특정 노드에 안 걸리고 공정 전반을 묻는 패턴 → 골격 트리+precedes 체인 통째를 그래프 사실로 공급(2단 얇은 골격이라 수십 노드 — 저비용).
- **답변 3단 규칙**: ⑴근거 있음 → 근거 기반 답 + 출처(청크ID/문서) 표기, 근거에 없는 내용은 모른다고 답함 ⑵근거 없음/부족 → "사내 문서에서 근거를 찾지 못했다"를 먼저 밝히고 **[일반지식 — 사내 검증 필요]** 표시와 함께 LLM 일반지식 답변 제공 + 등록 개체 안내("혹시 노칭·스태킹 중 어느 쪽 질문인가") ⑶링킹 미스는 로그로 축적(계기판 5번 재료). 일반지식 답이 사내 지식처럼 보이면 시스템 신뢰가 무너지므로 경계 표시는 계약.

**5.6.5 질문 유형 대응표 [골든셋 설계의 뼈대]**

질문이 요구하는 정보의 소재지는 네 곳뿐(엣지/노드 필드/청크/그래프 밖) — 이 기준의 전수 열거:

| # | 유형 | 예 | 답의 소재 | 지원 |
|---|---|---|---|---|
| Q1 | 개체 서술형 | "노칭 프레스 금형 관리 어떻게?" | 청크 | ✅ 기본 경로 |
| Q2 | 구조 조회형 | "노칭 다음은?", "노칭에 무슨 설비?" | 엣지 | ✅ 그래프 사실 |
| Q3 | 전역 구조형 | "전체 공정 흐름 설명해줘" | 엣지 전체 | ✅ flow 규칙 |
| Q4 | 값 조회형 | "노칭 정밀도 규격이 뭐야?" | 노드 필드 | ✅ 그래프 사실(필드) |
| Q5 | 역방향/필터형 | "클리어런스 관리하는 설비는?", "cathode 쪽 인자는?" | 역방향 엣지+필드 | ✅ 양방향 확장 + LLM 필터 (규모 커지면 코드 필터) |
| Q6 | 개체 없는 열린 질문 | "수율 올리려면 뭘 봐야?" | 청크(개체 경유 불가) | 훅만 — 미스율이 정당화 시 구현 |
| Q7 | 집계/분석형 | "관리인자 최다 공정은?", "심각도 높은 불량 top" | 그래프 연산 | 범위 밖 (플랫폼 대시보드) — 단 사내 빈출 질문일 수 있어 [질의 이연 목록]에 승격 후보로 기록 |
| Q8 | 그래프 밖 지식 | "노칭이 영어로?" | 없음 | ✅ 답변 3단 규칙 ⑵ |

**5.6.6 하이브리드 서치(Q6) 이연 결정**
- **임베딩 2종 구분 (P4 해석)**: (a) **판정용 노드 임베딩 [이연 아님 — v1.12, F16]** — canonical+정의문 기반, 수백~수천 개, 매 로드 시 on-the-fly 재생성 → P4의 "비저장"은 이것을 말함. (b) **검색용 청크 임베딩** — 청크 원문 기반, 수만~십만 개, 매 질의 재임베딩 불가 → **인덱스 캐시로 저장**하며, 이는 P4가 금지하는 것이 아니라 P5가 허용하는 파생물(청크 원문에서 언제든 재생성 가능).
- 이연 안전성 검증 완료: 필요 재료 중 임베딩 함수(core에 판정용으로 존재)·청크 저장(P5)·describes 역방향 조회·링킹 3단 훅은 기존에 있고, 신규는 청크 임베딩 인덱스 배치 생성 + top-k 함수 + 폴백 분기 1개뿐. 인덱스는 도입 시점에 청크 저장소에서 배치 재생성 가능 → "처음부터 구조를 맞출" 필요가 없음(좋은 원칙의 부산물로 이미 맞춰져 있음).
- **물리구현 사다리 (결정 기록 — 구현 아님)**: 도입 시 물리구현은 규모가 결정한다 — numpy 전수계산(청크 10만이어도 ~400MB·질의당 수십 ms, 당분간 충분) → faiss(수십만+) → 전용 vector DB(백만+ 또는 대규모 메타데이터 필터 필수). 셋 다 청크에서 재생성 가능한 파생물이라 교체 저비용. **초기값 = numpy 전수계산.**
- **전제조건 명문화: 전 청크 원문 보존(linked=False 포함)** — 폴백의 검색 대상은 전체 청크여야 열린 질문에 답이 됨. 부하는 미미: 저장 대상은 원본 바이너리가 아니라 파서가 뽑은 텍스트 조각(청크당 1~2KB, 문서 수천 건이어도 수십~수백 MB JSON). 원본은 source_path 참조만.
- 도입 판단 지표 = 링킹 미스율(계기판 5). 도입 시점 작업 = 검색 품질 튜닝(top-k·유사도 하한·tier 배치) — 언제 만들든 동일 비용이며, 미룰수록 실사용 미스 로그가 튜닝 데이터로 축적되어 유리.

## 6. 데이터 계약: 메타스키마 [확정 — 마감]

기준 계약 문서: **`role_어휘_정의서.md`** — role 어휘(필드 role 5종 + edges 스키마 선언)의 정의·판별 테스트·처리 규칙·스키마 예시(PFMEA/Control Plan)·등록 워크플로우의 **원본**은 정의서다. 본 절은 결정사항과 정의서에 없는 아키텍처 맥락을 담는다(요약이 아니라 역할 분담).

### 6.1 Role 메커니즘 [확정]
- **role = 필드 값으로 에이전트가 그래프에 하는 쓰기 동작.** 코드는 필드 이름이 아니라 role만 본다 → 필드 변경·새 doc_type은 스키마 파일 수정만으로 처리(코드 무변경).
- 구성: **필드 role 5종**(`anchor`/`entity`/`attribute`/`content`/`meta`, 각 필드 값에 붙어 핸들러가 처리) + **스키마 레벨 `edges` 선언**(role 아님 — 필드 해소 후 루프 후처리로 엣지 생성). "role 6종"이라 부를 때의 6번째는 edges를 가리키나, edges는 핸들러가 없는 선언이므로 코드상 5 핸들러 + edges 후처리로 구현한다.
- **봉쇄(완전성) 근거**: role 목록은 문서 다양성 예상이 아니라 **그래프 쓰기 동작의 전수 열거**(기존 노드 찾기/노드 만들기/필드 쓰기/엣지 만들기/청크 연결/무기록)에서 도출 — 제7의 동작은 그래프 모델상 존재하지 않으므로 커버리지가 경험이 아니라 구조로 보장된다. 이 보장이 깨지는 경우는 property graph 모델 자체가 못 담는 정보의 등장뿐이며 그것은 L4(별도 재검토)로 정의상 분리된다.

### 6.2 스키마 구조 [확정]
- **3층**: 문서 봉투(doc_id, doc_type, source_path, revision, parsed_at, parser_version, **context?** — 재인입 단위) + 조각 공통(chunk_id, doc_type, process_group, process_ref, process_no?, electrode_type, **context?**) + doc_type별 payload(prose | table).
- **context 상속 (v1.6)**: 봉투의 context({축:값} 임의 딕셔너리 — 예 {model:"A"})가 전 레코드의 기본 소속, 레코드에 context가 있으면 덮어씀(process_group/ref와 같은 상속 패턴). 축은 미리 발명하지 않음 — 확장은 스키마 파일(L1). 값은 스칼라 또는 리스트({line:[1,2,3]} = 공통 소속).
- 공통 조각 중 공정좌표 3형제(process_group/ref/no)의 본질 = "모든 정보 조각은 자기가 공정 트리 어디에 속하는지 이름표를 달고 들어온다"는 선언. **process_ref가 anchor이므로 부착이 LLM 추측이 아닌 결정적 사전 조회**가 된다(Tier1이 파서 출력 계약까지 거슬러 지배). 태깅이 골격과 안 맞으면 기존 경로(사전 미스→판정→orphan)가 받는다.
- electrode_type은 속성 차원이되 (v1.6 정정) **극성 결합 canonical을 통해 Unit·Property 노드 정체성에도 관여**(§5.2 entity 해소 규칙) — 단순 "질의 필터용"이 아님.
- prose(PPT/Word/PDF): 청크 계약(text/section/meta). 이미지 페이지는 파서가 LLM 요약을 text로 넣어 동일 형태로 수렴. 스키마가 사실상 "공통부+content" 고정형.
- table(Excel 표준양식): 행=record. **role 선언이 추출을 대체하므로 추출 LLM 불필요** — LLM은 개체 판정에만 남는다("정형=규칙+매칭, 산문=판단+추출" 원칙의 자동 구현).
- **블록(fragment) 조립**: 여러 doc_type에서 반복되는 필드 묶음을 블록으로 승격해 use_blocks로 조립. 승격은 관찰 후(스키마 레벨의 Rule of Three) — 단 공통코어·공정좌표는 전 문서 공통 확정이므로 첫날부터 블록. 블록도 값(B)이라 코드 리스크 0.

### 6.3 변경 등급 [확정]
| 등급 | 사건 | 고치는 것 | 코드 |
|---|---|---|---|
| L1 | 필드명 변경, 필드 추가/삭제 | 스키마 파일 한 줄 | 0 |
| L2 | 새 doc_type (기존 role로 표현 가능) | 스키마 파일 1개 추가 | 0 |
| L3 | 기존 role로 표현 불가한 필드 성격 | role 1종 + 핸들러 함수 1개 추가 | 함수 1개 (기존 무수정) |
| L4 | 그래프 모델이 못 담는 정보 | 설계 재검토 | 구조 변경 |
- **범용성의 정의: L1~L2가 사건의 대부분이 되게 하고, L3 비용을 함수 1개로 봉쇄하는 것.** L4는 어떤 설계로도 예방 불가한 영역임을 정직하게 기록. role 어휘를 "미래까지 완벽하게 지금 완성"하는 것은 범용성이 아니다(그것도 예측 게임) — 부족했을 때 확장이 국소적이도록 구조를 잡아두는 것이 확보 가능한 범용성의 정확한 정의.

### 6.4 role 어휘와 파서의 관계 [확정]
- **role 어휘 = 파서와 온톨로지 에이전트 사이의 계약 언어.** 파서에 대한 요구는 "예쁘게 파싱"이 아니라 "출력의 각 필드가 role 배정 가능한 형태일 것".
- 사슬: 사내 문서 →파서→ 메타데이터 JSON →role 스키마(번역표)→ 그래프 쓰기 동작 → 온톨로지.

### 6.5 스키마 연결 메커니즘 [확정]
"doc_type 스키마 ↔ 실제 문서 메타데이터"의 연결은 시점별 3단계로 분해되며 **별도 연결 에이전트는 불필요**(런타임 연결이 조회 1회이므로 과함):
1. **지금(외부)**: 문법 확정 — role 어휘, 스키마 파일 형식. ✅ 완료.
2. **등록 시점(사내, doc_type당 1회)**: 실물 문서를 보고 **기존 메타데이터 양식의 필드 → role 매칭** = 연결 행위 그 자체. 지능이 필요한 유일한 순간이며, 문서별 정형화 덕에 1회 번역표가 그 타입 문서 전부에 적용된다. **파서 출력 사양과 스키마 파일은 같은 자리에서 같은 문서를 보며 한 번에 확정** — 서로에게 맞추는 것이 아니라 둘 다 실물 문서에 맞추면서 "필드명 일치"라는 얇은 약속으로 묶인다. 필드명이 미리 나올 필요 없음: 메타스키마가 미리 강제하는 것은 공통부(봉투+공통조각)와 문법(role 배정 가능성)뿐.
3. **런타임(매 문서)**: doc_type 문자열로 스키마 파일 조회(dict 1회) → 기계 적용.
- **정합성 감시(인입 검증)**: 레코드에 스키마에 없는 필드 등장 → `unknown_field` 리뷰(파서가 앞서간 신호 → 스키마 갱신). 스키마에 있는데 레코드에 없음 → optional 검사. 어긋남이 조용한 오염 대신 큐에 시끄럽게 나타난다.
- 스키마 파일이 실질 필요한 것은 table(Excel) — 등록 부담 = Excel 표준양식 종수 × 1회.

### 6.6 처리 메커니즘 (핸들러 루프) [확정]
```python
HANDLERS = {"anchor": .., "entity": .., "attribute": .., "content": .., "meta": ..}
def ingest_record(record, schema, graph, dic):
    resolved = {}
    for field, spec in schema["fields"].items():
        resolved[field] = HANDLERS[spec["role"]](record.get(field), spec, graph, dic)
    for e in schema.get("edges", []):
        graph.add_edge(resolved[e["from"]], e["relation"], resolved[e["to"]])
```
- 위는 **개념 예시**(핵심 골격만). 실제 핸들러는 §7-6의 3단 구조(규칙 → LLM 폴백 훅 → 수정 큐)를 따르며, ingest는 2-pass(§5.5-1: Pass1 해소 → Pass2 부착·edges)로 구현한다. edges는 HANDLERS에 없고 루프 후처리(예시의 두 번째 for)로 처리 — 필드 role 5개 + edges 후처리.
- 새 role = 핸들러 함수 1개 작성 + 등록 1줄(기존 핸들러 무수정) — L3 봉쇄의 코드 형태.

### 6.7 사내 항목 [지정]
- [검증] 실물 2~3종(PFMEA, Control Plan 우선) 전 열에 role 배정 실험 — **사내 첫 작업은 코딩이 아니라 이 실험**. 배정 가능성은 §6.1로 보장되므로 검증 대상은 배정의 정확성(entity냐 content냐 등의 판단). 막히는 필드는 L3/L4 판정.
- [확정] doc_type별 스키마 내용, 블록 목록.
- 등록 워크플로우: 정의서+mock 스키마를 사내 LLM에 제시 → 실물 열 이름으로 초안 작성 → 사람 검수(필드당 6지선다 — role 오배정만 잡으면 됨) → 파일 저장. 코드 배포·수정 없음. LLM이 "어느 role에도 안 맞는다"고 답하면 그것이 검증 신호. 이 워크플로우가 플랫폼 "문서 타입 등록" 기능의 원형.

## 7. LLM 역할과 판정 메커니즘 [확정]

**역할 3분할: 사람이 구조를, 규칙이 관계를, LLM이 식별을.**

| 주체 | 결정하는 것 |
|---|---|
| 사람 | 골격(seed), 카테고리 체계·정의문, 관계 유형, 스키마(role·edges), 프롬프트 — 전부 값(B) |
| 규칙(코드) | 관계 선택(edges 선언·카테고리쌍 매핑), 사전 조회, 후보 검색, 임계 비교, 처리 순서 |
| LLM | 식별 — 추출(무엇이 언급됐나 + 닫힌 목록 분류), 개체 판정(같은 개념인가) |

1. **LLM 판정 지점은 두 곳뿐**: ①추출(prose 전용 — 청크 내용을 읽고 문맥으로 판단. 카테고리는 config의 닫힌 목록+정의문에서 선택, 목록 밖 발명은 구조적으로 차단 = Tier1의 프롬프트 버전) ②개체 판정(표면형이 기존 노드와 같은 개념인가 + 확신도 — 동의어 사전이 살찌는 지점).
2. **관계는 LLM이 결정하지 않는다**: table=스키마 edges 선언, prose=(카테고리 쌍→관계) 매핑, 골격=seed — 전부 사람이 쓴 값. LLM은 "무엇이(표면형·카테고리)"와 "어디에(부착대상)"까지만 → **관계 오류가 구조적으로 불가능**(LLM이 실수할 자유 자체가 없음).
3. **제어 흐름은 코드 소유**: 구축 파이프라인의 에이전트화(LLM이 다음 행동을 결정) 금지 — 실행마다 순서가 달라지면 재현·디버깅·감사(provenance)가 무너지고 P3가 깨진다. 파이프라인=코드가 순서 결정+LLM은 정해진 지점에서 정해진 질문에 답, 에이전트=LLM이 도구를 골라 루프 — 구축은 전자로 고정.
4. **에이전트화 허용 지점은 개체 판정 하나**: 어려운 케이스(후보 둘이 비슷: "프레스" → 노칭/탭용접 프레스)에서 읽기 전용 도구(이웃 조회·근거 청크 열람·사전 조회)를 쥔 판정 에이전트가 다단계 추론 가능. 범위가 판정 한 지점 + 도구 전부 읽기 전용이라 비결정성이 그래프를 오염 못 함. **도입은 보류율·판정 정확도 측정 후**(P7) — 효과가 "수정 큐로 가는 건수 감소"로 정확히 측정됨.
5. **판정 3분기**: 매칭(alias 자동 누적 — 사전이 크는 유일한 경로) / 신규(자동 생성, status=auto+provenance) / 불확실(**신규로 생성 + 수정 큐** — §9 "되돌리기 쉬운 쪽" 규칙).
6. **핸들러 3단 구조 (표준)**: 모든 role 핸들러는 `규칙 → LLM 폴백(훅) → 수정 큐` 3단. anchor/entity는 본래 이 구조(사전조회→판정→orphan). attribute/edges에는 **폴백 훅 자리만 확정** — 실제 채움은 사내 갭 관찰 후(P7). 관찰 대상 갭: 빈 셀 부착(부착 대상 부재), 상동/병합 셀, 복수값 셀("A, B"), entity 칸의 혼입 산문. 단 상동/병합/복수값의 1차 방어선은 파서의 자기완결 레코드 계약(§12).
7. **LLM 임무 봉쇄**: 파서 — 이해·이미지 요약·좌표 태깅 / 에이전트 — 추출·개체 판정 / 질의 — 답변 생성(④에서만, 그래프 직접 열람 없음). 좌표 오태깅은 anchor 미스→orphan 리뷰로 표면화(안전망).

## 8. 층간 연결 [확정 — v1.0 마감]

**연결 구조 (품질지식층 ↔ 공정층)**

```
[품질지식층]                          [공정층]
FailureEffect (골격, 셀 관점 분류)
    ▲ affects
Failure (발견형, 대다수)  ─── occurs_in ───▶  Process
    ↺ causes (층 내 연쇄)  ─ controlled_by ─▶  Property
```

1. **cross-layer 연결을 맡는 노드는 Failure 하나뿐.**
   - `Failure ─occurs_in→ Process`: 출처는 행의 공정좌표(process_ref, anchor). 정상 경로에서 모든 Failure의 **기본 닻**. **한 Failure는 여러 공정에서 발생 가능 → occurs_in 다중 엣지가 정상**(예: "이물 유입"이 노칭·실링 양쪽). 이것이 "이 불량을 낼 수 있는 공정들은?" 질의의 답이며, 병합의 대가가 아니라 이득(현상은 하나, 발생 공정은 여럿). 공정별 세부 구분이 정말 필요한 경우는 spec_conflict/사후 관찰로 표면화되어 node split. **앵커 미스 시**(process_ref가 골격에 없음) occurs_in은 생성되지 않고 orphan_anchor 큐가 그 Failure를 보유(orphan 재시도 배치의 대상) — "반드시"는 정상 경로 한정, 미스는 큐로 표면화. 이때 규칙 B의 폴백 대상(@process_ref)도 미스이므로 Property 부착도 함께 드롭됨(연쇄).
   - `Failure ─controlled_by→ Property`: 출처는 control_item 열. 행에 있을 때만 생기는 보조 연결. Property 부재 시 자동 생성 규칙(§15.7 규칙 A)이 받음.
2. **FailureEffect는 cross-layer 직접 엣지 없음.** 셀 관점 분류라 특정 공정에 붙지 않는다. 공정과의 관계는 경로로 표현: Effect ←affects← Failure ─occurs_in→ Process ("단락 유발 가능 공정은?" = 2홉 질의). 직접 엣지를 달면 경유 정보("어떤 불량을 통해")가 소실된다.
3. **Unit과의 직접 연결도 미도입** — controlled_by→Property→(has_property 역방향)→Unit 경로로 충분. PFMEA 실물에 설비 열이 확인되면 L1급 추가.
4. **생성 방향 규약: 품질지식층 → 공정층 단방향으로만 생성.** 공정층은 품질지식층의 존재를 모른다(층 독립성 — §3 단방향 의존의 그래프 형태). 질의 시 역방향 탐색은 엣지를 거꾸로 읽을 뿐 저장 방향과 무관.
5. **연결 시점·주체: 인입 자체가 실시간 연결.** 별도 연결 agent 불요 — 걸침 문서(PFMEA)의 필드별 category가 층을 지정하므로 한 번의 인입이 두 층에 동시 기록(§15.7 규칙 A). "걸침 문서의 소비 주체" 문제는 소멸.
6. ~~잔여~~ → **해소(v1.2)**: cross-layer 관계(occurs_in·controlled_by)는 질의 기본 확장에 **포함**하되 **1홉·비재귀·양방향 브리지**로 한정. 근거: cross 질의가 층간 연결의 존재 이유(PoC 가치)이며, 노이즈는 **채널 분리(그래프 사실은 답변 LLM이 질문 관련성으로 필터 §8-R4·§5.6.4) + 청크 tier2 잘림(§5.6.3)**으로 통제 — 1홉은 확장 깊이만 제한하지 딸림 자체를 막지 못함(라우터가 브리지를 상시 적용하므로 Failure는 매 질의 딸려옴). 파라미터 조정은 로그 관찰로(P7).
7. (장기) 기종(공정층) ↔ 개체 호기(설비이력층) 연결은 3번째 층 논의로 유지.
8. **3층 확장 자리표시**: 현 §8은 2층(품질→공정 단방향, Failure가 유일 연결 노드) 전제로 서술됨. 3번째 층(불량이력 등)은 공정·품질 양쪽에 연결될 수 있어 "단일 연결 노드·단방향" 전제가 확장될 수 있음 — §8-R 브리지가 다대다 cross-layer를 담는지는 3층 진입 시 실측(지금 코드는 관계 목록 순회라 개수 무가정, §3.6 — 구조는 열려 있음).

**8-R. read측 규약 (v1.2 추가)**
- **R1 라우터**: 질의 진입점은 cli/query.py **하나**(얇은 라우터). 절차 = 전역 링킹 → 해소 노드를 layer별로 묶어 각 층 config로 **core.query 범용 파이프라인** 호출 → cross-layer 브리지(위 6) 1홉 적용 → 그래프 사실+문서 근거 합성. 라우터·파이프라인 모두 제네릭 조립 코드이며(§3.4-(가) 준수·§3.6), 층별 확장 의미는 코드가 아니라 config.query_traverse가 소유. router는 층 폴더를 자동 발견(등록 코드 없음).
- **R2 사전 전역성**: 동의어 사전은 **전 층 공유 단일 저장**(data/dictionary.json). 계약: `lookup(surface) → 후보 노드 id 목록`(층 간 표면형 충돌 허용 — 호출자가 category/layer로 선별). canonical "층 내 유일" 규칙은 유지, 층 간 유일은 강제하지 않음.
- **R3 id 전역성**: 노드 id는 **층을 넘어 전역 유일**(전 층 공유 시퀀스). cross-layer 엣지가 타 층 노드 id를 직접 참조하는 §8 write 계약의 전제 조건을 명문화한 것.
- **R4 cross-layer 사실 문장화**: 브리지로 수집된 cross-layer 엣지는 해당 관계를 소유한 층(상위층)의 fact_templates로 문장화.

## 9. 수정 모델 [확정] — 구 "승인/수정"

**승인 게이트 폐기. 자동 커밋 + 사후 수정 모델.** ("흘려보내고, 표시하고, 나중에 고친다") — 사람이 매건 승인하는 것은 물량상 불가능하며, 파이프라인은 멈추지 않는다.

- 신규 entity 즉시 자동 생성. 모든 자동 생성물에 status(`auto`) + provenance(문서/행) 부착.
- 리뷰 큐 = 차단 대기열이 아니라 **수정 작업목록**. 사람은 자기 리듬으로 처리하며, 처리하지 않아도 시스템은 동작.
- **불확실 시 기본값 = 병합하지 말고 신규 생성.** 중복 노드 2개는 병합 도구로 쉽게 합치지만, 오병합은 동의어 사전 오염 + 발견·복구 곤란. **"되돌리기 쉬운 쪽으로 틀려라"** 가 수정 모델의 일반 규칙.
- **수정 도구 세트가 본체** (상세는 §13 항목): 노드 병합/분리, alias 이관, canonical 개명, 엣지 삭제, 큐 우선순위 정렬(참조 많은 auto 노드 우선 등), 근거 소멸 처리(§5.5-3).
- **spec 충돌 감지**: 기존 spec과 다른 값 유입 시 덮어쓰지 않고 충돌 큐 기록 — 개정 이력 관리가 자연 획득.
- **alias에도 provenance 기록**(출처 문서) — 오누적 발견 시 원인 진단(오판정 vs 실제 사내 혼용) 및 재인입 시 alias 정리 가능.

| 1 | 링킹 recall | 검색력 (골든셋 기반) |
| 2 | plateau 탐지 (문서별 신규 엔티티율 수렴) | 층 구축 완료 신호 |
| 3 | 판정 보류율 (수정 큐 유입률) | 판정 에이전트화 도입 판단 (§7-4) |
| 4 | 청크 잘림률 | 랭킹 고도화 도입 판단 (§5.6.3) |
| 5 | 링킹 미스율 | 하이브리드 서치 도입 판단 (§5.6.6) |

- 새 평가체계 설계는 후순위이나 위 5종은 한계비용이 낮고 **P7의 도입 판단 근거를 공급**하므로 유지. 골든셋은 질문 유형 대응표(§5.6.5)를 뼈대로 유형별 문항 구성. 측정 방법 상세는 §13 항목.

**질의 이연 목록 (저장은 하되 질의 대상 승격은 명세 확정 후 별도 논의)**
- is_cc / spec 구조체 내부 값 (§정의서 3.3) — CTQ 여부 등으로 필터 질의하려면 노드 레벨 태그로 승격 필요.
- RPN 인자(발생도/검출도 등, §15.5) — "심각도 N 이상" 류.
- 자재(§5.4) — 별도 층 후보.
- Q7 집계/분석("관리인자 최다 공정", "심각도 top") — 현재 플랫폼 대시보드 소관이나 사내 빈출 시 질의 승격 검토(그래프 집계 연산 필요).
- **context 크로스-노드 필터**("미국공장 3라인 flow만") — 필터 코드 자체는 저비용(딕셔너리 매칭)이나, 자연어에서 축값을 추출하는 질의측 링킹(축값 사전+프롬프트)이 본체라 이연. 특정 노드의 맥락형 값 표시(모델별 spec 나열)는 이연 아님 — 지금 됨.
- 이 목록의 승격은 각각 "노드 레벨 속성/태그로 빼기"를 요하며, 구조체 안에 묻힌 값은 검색되지 않음을 전제. 명세 확정(묶음3까지) 이후 일괄 검토.

## 11. 시각화 / 플랫폼 [확정]

- **진실 = data/의 JSON 그래프 + 청크 저장소.** Cypher export(Neo4j 적재)·Mermaid(보고)·임베딩 인덱스는 그래프에서 파생되는 재생성 가능물(P5). Neo4j 시각화는 플랫폼에 포함.
- 플랫폼↔파이프라인 연동은 파일 계약(subprocess) — 코드 의존 0.
- 플랫폼 핵심 기능의 원형: 스키마 등록 워크플로우(§6.7) + 수정 도구 세트(§9) + 계기판(§10) + Q7류 집계 대시보드.

## 12. 파서 계약 [확정]

파서는 사내 구현이며 내부에서 LLM을 어떻게 쓰든(이미지 요약, 청크 경계 판단, 셀 정돈) 자유. 본 시스템이 요구하는 것은 **출력 계약 준수**뿐:
1. 메타데이터 양식(봉투+공통조각+payload) 준수 — 각 필드가 role 배정 가능한 구조일 것.
2. 공통 좌표(process_group/ref 등) 태깅 의무 — 태깅용 닫힌 목록(골격 canonical+alias)은 시스템이 공급(골격 파일 = 공유 자산).
3. **레코드 자기완결성**: 상동("〃") 해소, 병합 셀 전개, 복수값 분리 완료 — 행 단독으로 의미가 성립할 것. **표면형에 극성 미결합**(극성은 electrode_type 필드로만 — 결합은 에이전트 몫 §5.2, 이중 접두 방지, v1.12).
- 인입 검증(§6.5)이 계약 위반을 unknown_field/optional 검사로 감시.

## 15. 품질지식층 정의 [확정 — v1.0, 여섯 칸 마감]

### 15.1 성격
PFMEA류가 담는 **정적 품질 지식**(잠재 고장모드/원인/영향의 체계). 실제 발생 사건(불량이력층, 3번째 후보)과 구분. 입력은 table(PFMEA)이 주 — prose 추출 경로는 훅만.

### 15.2 칸2 — 골격
- **FailureEffect의 얇은 Tier1 목록** = 셀 관점 고장영향 분류(단락, 화재, 방전기능상실, 충전기능상실 등 — 실물 목록은 사내 확정). 안정적·공정무관·유한하므로 골격 자격 충족.
- 공정층 골격=트리, 품질지식층 골격=**평평한 목록** — 골격 심기 코드가 층마다 다름의 실측 표본(Rule of Three 관찰 재료).
- 불량 간 인과("이물→외관/저전압")는 골격이 아니라 **발견되는 causes 엣지**(분류 체계가 아니라 인과 지식이므로). 문서 근거 없는 도메인 지식은 사람이 넣는 Tier1 seed 엣지 옵션으로만.

### 15.3 칸3 — 관계
| 관계 | 연결 | 성격 |
|---|---|---|
| `causes` | Failure→Failure | 층 내 인과 연쇄 (방향은 행의 열 구분+edges 선언이 부여) |
| `affects` | Failure→FailureEffect | 층 내, 골격 분류로의 귀결 |
| `occurs_in` | Failure→Process | cross-layer 기본 닻 (§8) |
| `controlled_by` | Failure→Property | cross-layer 보조 (§8) |

### 15.4 칸4 — 카테고리 정의문
> **Failure(불량/고장 현상)** — 공정에서 발생할 수 있는 불량·고장 현상. PFMEA의 고장모드와 고장원인을 **병합**한 열린 집합(같은 현상이 어떤 행에선 원인, 다른 행에선 모드로 등장 — 한 노드로 해소되어야 causes 연쇄가 이어짐). 방향 정보는 카테고리가 아니라 행 구조가 보유.
> **FailureEffect(고장영향)** — 셀 관점의 결과 분류(단락, 화재 등). 닫힌 Tier1 목록, anchor로 조회됨. 신규 표기는 orphan 리뷰(유형은 사람 — P2).
- 병합의 한계 기록: prose에서 Failure–Failure의 방향은 (카테고리쌍→관계) 매핑으로 결정 불가 — PoC는 table이라 무영향, prose 확장 시 사내 케이스로 판단([사내 항목]).

### 15.5 칸1 — PFMEA 스키마 (role 배정, 사내 실물로 교정)
nested(모드 1 : 원인 N)는 파서가 모드×원인 쌍으로 전개(자기완결 레코드, §12-3).
| 필드 | role |
|---|---|
| 공통좌표(process_ref 등) | anchor |
| failure_mode / cause | entity(Failure) |
| severity | attribute → **effect_category** (v1.3 정정) |
| effect.category | **anchor**(FailureEffect 골격 조회) |
| effect.detail | content 또는 attribute ([사내 확정]) |
| control_item_for_fm / _for_cause | entity(Property, **공정층**) — 걸침 필드 |
| prevention_control / detection_control | content — describes 대상 = 행의 cause / failure_mode (table은 행 문맥이 명확하므로 LLM 추출 없이 attach_to_field로 직접 지정) |
edges: cause─causes→failure_mode / failure_mode─affects→effect.category / failure_mode·cause─controlled_by→control_item / failure_mode─occurs_in→process_ref.
- **severity 부착 정정(v1.3)**: severity는 failure_mode가 아니라 **effect_category(FailureEffect)** 에 부착. 표준 FMEA에서 S는 영향(effect) 기준값이므로 영향이 같으면 S 동일 → Failure 병합 노드의 우발 spec_conflict가 구조적으로 소멸. mock은 effect_category ↔ severity를 1:1 정렬할 것.
- **RPN 인자 이연(v1.3)**: 행 단위 RPN 인자(발생도/검출도 등)의 정주소는 노드가 아니라 causes 엣지 속성(엣지가 모드×원인 행의 실체)이나, 현 실물 스키마엔 severity만 존재하고 영향 단위 저장으로 충분. RPN 기반 질의(심각도 N 이상 등)는 **PoC 밖 — 저장만, 이연**. 발생도/검출도가 실물에 등장하거나 RPN 질의가 요구되면 엣지 속성 도입 결정(P7).

### 15.6 칸5·6 — 절차
- 쓰기: 공정층 절차 복사+수정(Rule of Three 실행). 2-pass·임계 0.85·재인입·orphan 배치 동일 적용.
- 읽기: query_traverse에 causes·affects 추가. cross_layer_traverse(occurs_in·controlled_by)는 **기본 포함 확정**(1홉·비재귀·양방향 — §8-6 v1.2에서 이연 해소, 노이즈는 채널분리+tier잘림으로 통제).

### 15.7 걸침 문서 처리 규칙
- **규칙 A**: PFMEA의 관리항목 필드는 공정층 Property로 인입 — 없으면 자동 생성(auto+수정 큐). PFMEA는 공정층의 보조 공급자(주 공급자는 CP). 소비 주체 라우팅 불요 — 필드 category가 층을 지정, 한 인입이 두 층에 기록.
- **규칙 B — 부착 폴백**: auto Property는 행의 공정좌표(process_ref)에 우선 부착. CP가 더 정밀한 소속(설비)을 주면 수정 큐/재해소로 보강 — "되돌리기 쉬운 쪽"(공정 직결은 오류가 아니라 저해상도).

### 15.8 사내 확인 항목
FailureEffect 표준 목록 실물 / effect.detail 성격(attribute vs content) / Failure–Failure prose 방향 케이스 / PFMEA에 설비 직접 지목 열 존재 여부(있으면 §8-3 L1 추가).

## 16. 구현 진입 방식 [확정 — v1.1, ④ 마감]

### 16.1 파이프라인 우선 (플랫폼 직행 배제)
- 근거: 진실(JSON 그래프)이 먼저 서야 플랫폼이 보여줄 것이 생김 / 핵심 검증(인입·질의 4단·Rule of Three)은 CLI+mock이 최속 / P1 덕에 선(先)파이프라인이 재작업을 만들지 않음.
- **플랫폼화 인지 계약 2개** (파이프라인 구상 시 준수):
  1. 모든 단계는 **CLI 진입점 + 파일 입출력**으로 노출 (`parse → build → query`) — 플랫폼은 이를 subprocess로 호출.
  2. 파이프라인은 그래프를 수정하지 않고 **수정 재료를 빠짐없이 기록** (status=auto, provenance, 수정 큐) — 큐 소화는 플랫폼/수정 도구의 몫. 재료가 온전하면 수정 UI는 언제든 부착 가능.
  3. **build 직렬 계약 (v1.14, F14)**: build는 **직렬 실행**을 전제한다 — id 발급(전역 시퀀스)·저장(graph.json 전체 재기록)이 프로세스 간 원자적이지 않으므로, 동시 build는 id 충돌(P4 유일성 위반)·저장 유실을 낳는다. 플랫폼이 파이프라인을 subprocess로 호출할 때 **호출부가 직렬화를 보장**해야 한다(한 번에 하나의 build). **개별 파일의 원자적 쓰기(tmp+os.replace)는 지금 구현**(쓰다 만 파일 방지) — 프로세스 간 파일 락만 단위 4로 이연. 계약은 지금 확정 — 이는 성능(P7 이연)과 별개인 정합성 요구다.

### 16.2 기존 platform 재활용 범위
- 재활용: subprocess 파일 계약 패턴, 워크스페이스 골조, 그래프 뷰어, Neo4j 시각화. 기존 승인 워크스페이스는 §9 수정 모델 구현의 **참고 원형**.
- 개조 4포인트: ①데이터 계약 교체(신규 JSON, 2층+cross-layer 표시) ②승인→수정 큐 의미 전환(§9) ③스키마 등록 워크스페이스(후순위 — PoC는 파일 직접 작성) ④계기판(③과 함께, 최후순위).

### 16.3 mock 데이터 정의
- **목적 = 메커니즘 검증으로 한정.** 품질 측정은 mock으로 불가·유해(가짜 데이터의 점수는 가짜 확신) — 국면 2-5(실데이터 파일럿)의 몫. 데모도 실데이터로.
- 설계 원칙: **모든 mock 항목은 검증할 메커니즘을 최소 하나 짊어진다** (행별 주석으로 명시).
- 사내 유래 정보는 모양만(공정 단계명·고장영향 분류·열 이름 — 업계 일반 수준), 내용물은 전량 창작.
- 규모: 골격 seed(공정 7노드 + FailureEffect 4~6) / PFMEA 12행(병합 연쇄·규칙A·규칙B·orphan·표기 변형 케이스 포함) / CP 6행(규칙B 보강 경로) / PPT 청크 8개(주제 vs 스침 함정) / 질의 12문항(Q1~Q5·Q8 × 2, cross-layer 포함 — ③ 골든셋과 별개인 스모크용).
- USE_MOCK 토글: LLM 없이 규칙 폴백으로 전 파이프라인 구동 (검수 속도 확보).

### 16.4 문서 체계 (⑤의 산출물 위치)
- **3문서 체계**: 명세(본 문서 — 무엇을·왜, 정본) / role_어휘_정의서(계약) / **구현용 문서(어떻게 — Claude Code 입력물, 별도 파일)**.
