## 첫 지시 (붙여넣기용)

```
너는 이 레포에서 제조 온톨로지/GraphDB 파이프라인을 구현한다. 아래 세 문서가 완전한 명세다:
- docs/ontology_system_명세.md   ← 진실(뼈대). 충돌 시 최우선.
- docs/role_어휘_정의서.md        ← role 계약.
- docs/ontology_system_구현용문서.md ← 파일트리·JSON계약·config전문·mock전문·구현순서(§7)·실행절차(§9).
우선순위: 명세 > 정의서 > 구현문서. 세 문서로 결정 안 되는 것은 추측하지 말고 §9.4를 따른다.

[작업 방식 — 자율 루프]
구현문서 §9.2의 착수 단위를 순서대로(1a→1b→1c→1d→2→3→…) 하나씩 진행한다. 각 단위마다:
  1) 해당 단위 코드를 생성한다. 구현 불변 규칙(구현문서 §0)을 절대 위반하지 않는다.
  2) 그 단위의 통과 조건(§9.2 표·§7 완료판정)을 tests/test_<단위>.py 로 구현한다(assert 기반, USE_MOCK=1).
  3) 테스트를 실행한다. 실패하면 코드를 수정하고 다시 실행한다. 통과할 때까지 반복한다.
  4) 통과하면 PROGRESS.md를 갱신하고 다음 단위로 넘어간다.
이 루프를 사람 개입 없이 자율로 계속 돌려라. USE_MOCK=1이라 네트워크·API키 없이 전 과정이 돌아야 한다.

[막히면 — 추측 금지]
문서로 판단이 안 서는 지점을 만나면 절대 임의로 메우지 마라. 대신 §9.4대로:
  - BLOCKERS.md에 [단위/파일/무엇이 불명확/어떤 결정 필요]를 기록하고,
  - 그 지점을 건너뛰고 의존 없는 다음 독립 작업으로 이동한다(막힌 것 위에 추측을 쌓지 않는다).
  - config로 표현 안 되는 층별 절차(명세 §3.6)는 raise로 명시적 실패시키고 BLOCKERS.md에도 남긴다.
금지: 명세와 다른 임의 결정으로 빈틈 메우기 / §0 불변 규칙 위반으로 우회 / 판정에 맞추려 테스트를 느슨하게 하기.

[시작]
먼저 다음을 하고 시작하라:
  a) 세 문서를 통독하고, 이해한 아키텍처를 5줄 이내로 요약해 확인시켜라(내가 방향을 점검한다).
  b) 레포 스캐폴딩: 구현문서 §1 파일트리대로 디렉토리·__init__.py·requirements.txt·.gitignore·빈 data/ 초기화(§9.1)를 만든다.
  c) 그다음 단위 1a부터 자율 루프를 시작한다.
각 단위 완료 시 PROGRESS.md에 [완료 단위/테스트 결과/생성 파일/git diff core 상태]를 남겨라.
단위 3 완료 시 `git diff core/` 가 비어 있는지 반드시 확인하라 — 이것이 config-only 확증(명세 §3.6)이며 핵심 성공 판정이다.
```

```
PROGRESS.md와 BLOCKERS.md를 먼저 읽어라. 마지막 통과 단위 다음부터 자율 루프를 재개한다.
BLOCKERS.md에 항목이 있으면, 내가 지금 답해줄 것을 먼저 물어라(그것부터 풀고 재개).
없으면 바로 다음 단위로 진행하라. 작업 방식·막힘 프로토콜은 이전과 동일(구현문서 §9).
```

## 사람(너)의 검수 체크포인트

각 창 종료 후 네가 볼 것 — 길게 안 봐도 되게 정리:
1. **BLOCKERS.md** — 제일 먼저. 추측을 피하고 멈춘 지점 = 네 결정이 필요한 곳. 여기 답해주는 게 네 핵심 역할.
2. **PROGRESS.md** — 어디까지 통과했나, 테스트 결과.
3. **단위 3의 git diff core/** — 비어 있으면 config-only 성공(품질층이 코드 없이 config만으로 돌았다는 실측). 비어있지 않으면 어느 결정점이 config로 표현 안 됐는지가 3층 대비 관찰 데이터.
4. **data/*/graph.json·review_queue.json** 눈으로 훑기 — mock이 의도대로 노드·엣지·큐를 만들었나(특히 C4 spec_conflict, C8/C9 mirrors+mirror_asymmetry, R9/R13 orphan).
