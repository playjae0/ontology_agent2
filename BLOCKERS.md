# BLOCKERS

> 자율 실행 중 문서(명세 > 정의서 > 구현문서)로 **결정 안 되는 지점**을 만나면 여기에 기록하고,
> 그 지점을 건너뛰어 의존 없는 다음 작업으로 이동한다(추측으로 메우지 않음 — 구현문서 §9.4).
> 각 항목: [단위 / 파일·줄 / 무엇이 불명확한지 / 어떤 결정이 필요한지].

## 단위 4 — 플랫폼 연동 (착수 보류, 사람 결정 필요)
- **무엇이 불명확**: 구현문서·명세엔 플랫폼 연동을 §16.2 "개조 4포인트" 개괄로만 서술(데이터 계약 교체·승인→수정큐 의미전환·스키마 등록 워크스페이스·계기판). 기존 `../platform/` 폴더의 실제 구조·엔트리포인트·데이터 계약이 세 문서에 없음.
- **필요한 결정**: (a) 기존 `platform/`(Ontology/platform)을 개조 대상으로 삼을지, 아니면 신규 최소 뷰어를 만들지. (b) 만든다면 어느 화면(2층+cross 그래프 표시·수정 큐 열람)부터인지. (c) subprocess 계약(`python -m cli.build` / `cli.query`)은 이미 준비됨 — 플랫폼이 이걸 호출.
- **왜 멈춤(§9.4)**: platform 실물 구조를 추측으로 개조하면 §1(파일계약) 위반 위험 + 재작업. 사람이 platform 범위를 지정해야 자율 진행 가능.
- 참고: 파이프라인 측 플랫폼화 계약(§16.1)은 이미 충족 — 모든 단계가 CLI+파일 입출력, 자동커밋+provenance+수정큐로 재료 온전.

## 단위 5 — 수정 도구 세트 + 계기판 (선행 미마감)
- **무엇이 불명확**: §7 표·§13에 따르면 단위 5는 "명세 ②(수정 도구 세트 상세) 마감 후 착수". 명세 §9는 도구 목록(병합/분리, alias 이관, canonical 개명, 엣지 삭제, 큐 우선순위, 근거소멸 처리)만 있고 각 도구의 입출력 계약·CLI 형태가 미확정(②는 사내/후속 논의 대기, §13 미결표).
- **지금 준비된 것**: 엣지 삭제 툼스톤(`status="deleted_by_user"`) 계약은 graph.add_edge가 이미 준수(재인입 부활 방지). enforcement 강화·나머지 도구는 이 단위 착수 시 구현.
- **필요한 결정**: ② 마감(각 수정 도구의 CLI 계약) 또는 "우선 엣지삭제/병합/개명 3종만 구현" 같은 범위 지정.
